AI agents are getting their own wallets. They are going to spend money autonomously — paying for APIs, buying compute, subscribing to data feeds, and transacting on-chain over the next decade.

The Open Wallet Standard (OWS), pioneered by MoonPay, is building the infrastructure for this reality. It provides a secure, universal way for agents to interact with blockchain wallets without ever holding the private keys. But as we transition into this agentic economy, a critical question arises: Who controls what the agent can spend, and how?

Right now, the answer is developers writing code.

The Problem with Config Files

OWS ships with a pre-signing policy engine. Before any key is touched, the engine evaluates the requested transaction against configured rules. It works perfectly, but it is rigid. A standard spending policy today looks like this:

{ "max_spend_usd": 20, "allowed_chains": ["base"], "blocked_days": [0, 6] }

If an operator wants to change a spending limit from $20 to $50, they have to file a ticket. A developer edits a config file, and it goes through a deployment cycle.

Furthermore, config files are binary — they completely fail on nuance. You cannot easily write a JSON rule that says: "Never spend more than $20 on a single transaction, unless it's for emergency cloud infrastructure."

If the whole point of AI agents is autonomy, the rules governing that autonomy shouldn't be locked behind rigid technical implementation.

Enter SENTINEL: The Natural Language Policy Engine

For the OWS Hackathon, we built SENTINEL — an extension to OWS that replaces config files and code with plain English.

With SENTINEL, your agent's spending policy is literally a document:

AGENT SPENDING POLICY
1. Only pay for API services, data feeds, and cloud infrastructure.
2. Never spend more than $20 on a single transaction.
3. Total daily spending must not exceed $50.
4. If the agent goes entirely silent for 7 days, trigger the dead agent switch and send all remaining funds to the backup wallet.

Every time the agent wants to spend, it gets stopped by the policy engine. But instead of parsing a JSON file, the engine bundles the transaction details with your plain text policy and hands them to Claude (LLM).

Claude evaluates the intent of the transaction against the English rules. It responds with an APPROVED or REJECTED decision, citing the exact rule it followed. Only a definitive "Yes" from Claude unlocks the wallet to call ows_sign.

Nuance, Speed, and the Dead Agent Switch

This architectural shift unlocks three massive benefits:

ACCESSIBILITY
The person who understands the business rules — the founder, the treasury manager — can write and edit the policy themselves. Change a sentence, hit save, the agent updates on the next transaction.
CONTEXTUAL NUANCE
A config file blocks $20.01 universally. Claude understands intent. "Only on weekdays, but allow weekends if our server goes down" is a completely valid rule SENTINEL can enforce without complex logic trees.
DEAD AGENT SWITCH
By simply writing "If inactive for 7 days, send funds to backup" in the policy document, the engine's background heartbeat safely executes an auto-recovery transaction. No secondary monitoring system needed.

Extending the Primitive

SENTINEL isn't a wrapper around OWS — it's an extension of its most important primitive. The policy engine is the soul of the standard. It is what stands between an autonomous machine and your money.

By upgrading that engine to understand natural language, we ensure that as AI agents become more sophisticated, our ability to control and guide them remains human.
